# 开题报告

**reL4 中 `sel4_cspace` 的组件化重构与 capability 核心原语模块级验证研究**

## 一、选题依据

### （一）选题背景

随着操作系统在安全关键场景中的应用不断增加，系统软件的安全性、可靠性和可验证性越来越受到重视。相比宏内核，微内核由于代码规模较小、结构相对清晰，更适合作为形式化验证的研究对象。seL4 作为微内核形式化验证领域的代表性成果，已经证明了对关键内核模块进行严格建模和验证是可行的，这也为后续相关研究提供了重要参考。

在微内核中，capability 机制是实现资源访问控制和安全隔离的重要基础。capability 的派生、插入、移动、删除、撤销，以及与对象创建过程相关的插槽更新，都会直接影响系统状态的正确性和安全性，因此 capability 相关代码具有较高的研究和验证价值。

reL4 最初是一个面向 seL4 Rust 重写的毕业设计项目，后续 reL4 team 和相关社区在此基础上继续做了一些整理和演进。就 capability 子系统而言，当前实现已经具备一定的组件化基础：`sel4_cspace` 中不仅包含 `cte` 和 capability derivation tree 的维护逻辑，也已经承载了 `derive_cap`、`delete_all`、`revoke`、`cte_insert`、`cte_move`、`cte_swap`、`insert_new_cap`、`resolve_address_bits` 等一批 CSpace 核心原语。与此同时，kernel 仓库中的 syscall invocation 路径仍保留了 `Copy`、`Mint`、`Move`、`Mutate`、`Rotate`、`Delete`、`Revoke` 等系统调用级 capability 操作的包装逻辑，这部分代码主要负责参数检查、错误处理、线程状态切换以及对 `sel4_cspace` 原语的组织调用。另外，`Retype` 目前仍位于 untyped invocation 路径中，它既涉及 capability 插入，也涉及对象创建和 untyped 状态更新。

因此，当前 reL4 capability 子系统面临的问题并不是 capability 逻辑完全没有进入 `sel4_cspace`，而是 syscall 层编排逻辑、untyped 管理逻辑与 cspace 核心原语之间的边界还不够清楚。如果直接在现有结构上对 `sel4_cspace` 开展形式化验证，会遇到验证对象边界不够稳定、抽象模型不容易统一、证明职责不够明确等问题。基于这一现状，本文打算先结合当前仓库实现，对 capability 子系统的边界做进一步整理，把适合归入 CSpace 原语的 capability 逻辑继续下沉到 `sel4_cspace`，然后在相对清晰的模块边界上开展抽象建模和模块级验证。

### （二）研究目的

本课题面向 reL4 capability 子系统当前“已经初步组件化，但边界还不够清晰”的现状，进一步梳理 syscall invocation、untyped 管理逻辑与 `sel4_cspace` 之间的职责分布。研究的一个重点是把仍适合下沉为 CSpace 原语的 capability 操作逻辑，从 kernel 中的 syscall invocation 路径进一步整理并下沉到 `sel4_cspace`，使 cspace 模块的职责更加集中，接口更加清晰。

在此基础上，本文将建立 `sel4_cspace` 的抽象状态模型和关键不变式，并围绕 `Copy`、`Mint`、`Move`、`Delete`、`Revoke` 等 capability 核心原语设计抽象规约；对于 `Retype`，则重点关注其中和 cspace 直接相关的 capability 生成、插入及派生关系维护部分。最后，尝试基于 Verus 对重构后的关键实现开展模块级形式化验证。

### （三）研究意义

本课题的研究意义主要体现在以下几个方面。

首先，从工程角度看，本课题有助于进一步理清 reL4 capability 子系统的模块边界。当前 `sel4_cspace` 已经承担了一部分 CSpace 核心原语，但系统调用级 capability 包装逻辑和部分相关职责仍分布在 kernel invocation 路径中。对这些边界进行进一步整理，可以减少 capability 逻辑在多个模块之间的分散，提高模块职责的清晰度，也为后续维护和扩展打下基础。

其次，从验证角度看，本课题能够为 `sel4_cspace` 提供一个更适合建模和证明的对象边界。相比直接在当前混合结构上开展验证，先把 capability 核心原语、系统调用级编排逻辑和 untyped 对象管理逻辑区分清楚，再围绕相对稳定的 cspace 边界建立抽象模型与规约，更有利于控制验证复杂度，也更容易形成较稳定的抽象语义。

最后，从方法上看，本课题不仅关注模块级形式化验证本身，也关注“怎样先把组件边界整理到更适合验证的状态”。这对 Rust 微内核后续演进也有一定参考意义，即形式化验证不只是事后附加的证明工作，也可以反过来推动模块职责和组件边界的优化。

### （四）国内外研究现状及发展动态

在微内核形式化验证方面，seL4 的相关研究已经比较成熟，通常以抽象模型、状态不变式、操作规约和精化关系为主线，论证关键模块在抽象层和实现层之间的一致性。这些研究说明，对微内核核心机制进行严格验证在理论和工程上都是可行的。

在 Rust 程序验证方面，近年来相关工具发展较快。Verus 等工具强调“规格靠近实现”，可以在现有 Rust 代码基础上补充前置条件、后置条件和辅助引理，比较适合进行模块级和局部性的验证工作。这一点对 capability 子系统尤其重要，因为这类模块内部关系复杂，但在边界较清晰的情况下，更容易组织抽象模型和证明结构。

不过对于 reL4 来说，当前更现实的问题并不只是“如何验证”，还包括“验证对象本身是否已经具备合适的组件边界”。现有 capability 相关逻辑虽然已经有一部分进入 `sel4_cspace`，但 syscall invocation 路径中仍保留着系统调用级 capability 操作包装，而 `Retype` 又与对象创建和 untyped 资源管理紧密耦合。这意味着在开展验证之前，有必要先结合当前实现，对 capability 相关逻辑继续做面向验证的整理。因此，本课题的重点不只是直接给现有代码补规格和证明，也包括在现有基础上进一步形成更适合验证的 `sel4_cspace` 边界。


## 二、研究目标和内容

### （一）研究思路

本课题拟按照“结构梳理—边界澄清—组件化重构—抽象建模—规约设计—视图桥接—实现验证”的思路展开。

首先，对 reL4 中 capability 子系统的现有实现进行分析，梳理 `sel4_cspace`、`cte`、权能派生树以及 syscall invocation、untyped invocation 路径中 capability 相关逻辑的职责分布，明确哪些逻辑已经属于 cspace 核心原语，哪些逻辑属于系统调用级编排，哪些逻辑应继续保留在对象创建和 untyped 管理路径中。

其次，在当前仓库实现基础上，针对仍适合整理为 CSpace 原语的 capability 操作逻辑进行组件化调整，将其从 syscall invocation 路径中进一步抽离并下沉到 `sel4_cspace`，使 cspace 模块边界更加清晰。

然后，在重构后的模块基础上，建立 capability、slot、对象、权限和派生关系等抽象表示，构造 cspace 的抽象状态模型及关键不变式，并重点围绕 `Copy`、`Mint`、`Move`、`Delete`、`Revoke` 等操作设计抽象规约；对于 `Retype`，重点描述其中与 cspace 直接相关的 capability 生成、插入及派生关系更新语义。

最后，建立 concrete state 到 abstract state 的 `view`，并在 Verus 中对重构后的核心实现开展模块级验证。

### （二）研究内容

本课题的主要研究内容包括以下几个方面。

第一，分析 reL4 capability 子系统的现有结构，梳理 `sel4_cspace`、syscall invocation 与 untyped invocation 之间 capability 相关逻辑的职责分布，明确当前组件边界、耦合关系及其对建模与验证造成的影响。

第二，对 capability 操作中适合整理为 CSpace 原语的逻辑进行组件化重构，将其进一步归入 `sel4_cspace`，形成更适合后续建模与验证的模块边界。

第三，在重构后的 `sel4_cspace` 基础上，建立 cspace 抽象状态模型，并设计 capability、slot、对象、权限和派生关系等基本语义。

第四，建立 cspace 的关键不变式，并重点为 `Copy`、`Mint`、`Move`、`Delete`、`Revoke` 等 capability 核心原语设计抽象规约；对于 `Retype`，重点描述其中与 cspace 直接相关的 capability 插入和派生关系维护部分。

第五，建立 concrete / abstract `view`，并在 Verus 中对部分核心实现进行模块级形式化验证，证明其满足抽象规约并保持关键不变式。

### （三）关键问题

本课题拟重点解决以下几个问题：

1. 当前 reL4 capability 子系统中，哪些逻辑应归入 `sel4_cspace` 的核心原语，哪些逻辑仍应保留在 syscall invocation 的系统调用级编排层，哪些逻辑应继续保留在 untyped 对象管理路径中；
2. 如何在当前“部分组件化”的基础上，为 `sel4_cspace` 建立职责清晰且适合验证的模块边界；
3. 如何基于重构后的 cspace 组件建立抽象状态模型和关键不变式；
4. 如何形式化描述 `Copy`、`Mint`、`Move`、`Delete`、`Revoke` 等核心 capability 原语，以及 `Retype` 中与 cspace 直接相关部分的抽象语义；
5. 如何将重构后的具体实现状态映射为抽象模型中的关系，并完成若干模块级证明。

## 三、研究方案

### （一）研究方法

本课题主要采用以下几种方法。

一是文献调研与理论分析。通过阅读微内核形式化验证、capability 机制、Rust 程序验证和 Verus 相关资料，梳理已有研究方法和可借鉴思路。

二是源码分析。围绕 reL4 capability 子系统，特别是 `sel4_cspace`、syscall invocation 路径和 untyped invocation 路径中的 capability 相关实现，分析其职责分布和当前边界。

三是组件化重构。依据 capability 操作的职责边界，将其中适合归入 cspace 核心原语的逻辑进一步下沉到 `sel4_cspace`，形成更清晰的组件边界。

四是抽象建模与规约设计。在重构后的模块基础上，建立 cspace 抽象状态模型、关键不变式以及核心操作规约。

五是实现验证。基于 Verus，对重构后的部分核心实现补充形式化规格，并尝试完成若干模块级证明。

### （二）技术路线

本课题的技术路线为：首先完成 reL4 capability 子系统的结构梳理，分析 `sel4_cspace`、syscall invocation 与 untyped invocation 之间 capability 逻辑的分布情况；其次，对其中适合整理为 CSpace 原语的逻辑进行进一步下沉和组件化重构；然后在新的 cspace 边界基础上建立抽象模型和关键不变式；接着设计 `Copy`、`Mint`、`Move`、`Delete`、`Revoke` 等核心 capability 原语以及 `Retype` 中 cspace 相关部分的抽象规约；最后构造 concrete state 到 abstract state 的 `view`，并在 Verus 中完成若干模块级证明。

### （三）可行性分析

本课题具有一定可行性。

首先，当前 `sel4_cspace` 已经包含 `cte` 操作、权能派生树维护以及若干 capability 核心原语，这为后续进一步整理 capability 边界和开展验证提供了较好的基础。相比从零开始构造验证对象，本课题面对的是一个已经初步组件化的 cspace 模块。

其次，本课题并不是以整个 capability 子系统为对象进行一次性彻底重构，而是以 `sel4_cspace` 为中心，只针对适合整理为 CSpace 原语的部分进行调整，同时把系统调用级编排逻辑和 untyped 对象管理逻辑保留在相邻层次中，因此重构范围相对可控。

再次，在完成边界整理之后，`sel4_cspace` 的模块职责会更加集中，抽象模型和验证边界也更容易明确，这有利于后续在 Verus 中开展模块级验证。

当然，本课题也存在一定难点。例如 capability 核心原语与 syscall 层包装逻辑之间的边界划分并不总是十分明确，`Retype` 的 capability 插入语义与对象创建语义之间也需要谨慎分离；另外，重构后的接口设计还要兼顾现有代码依赖，而 Verus 证明本身也可能带来较大的工程工作量。对此，本课题将优先从代表性操作入手，采取“先澄清边界，再建模规约，最后逐步证明”的策略推进工作。

## 四、研究计划及进度安排

1. 阅读 reL4 capability 子系统相关源码，梳理 `sel4_cspace`、syscall invocation 与 untyped invocation 路径中的 capability 逻辑分布；
2. 跑通最小 Verus 样例，熟悉基本验证流程；
3. 分析当前 capability 操作的职责边界，明确 cspace 核心原语、系统调用级编排逻辑与 untyped 管理逻辑的划分；
4. 完成 capability 相关逻辑向 `sel4_cspace` 的初步下沉与组件化整理；
5. 建立重构后 `sel4_cspace` 的结构视图与接口边界说明；
6. 建立 capability、slot、派生关系等基础语义，形成抽象模型初稿；
7. 完成关键不变式草案；
8. 设计 `Copy`、`Mint`、`Move`、`Delete`、`Revoke` 等核心原语的抽象规约，并补充 `Retype` 中 cspace 相关部分的规约；
9. 建立 concrete state 到 abstract state 的 `view`；
10. 给部分核心实现补充 Verus 规格，并完成若干模块级证明；
11. 完成论文撰写、答辩 PPT 和演示材料准备。

2. 在现有仓库分层基础上，将 capability 操作中适合整理为 cspace 核心原语的部分从 syscall invocation 路径中进一步抽离并下沉到 `sel4_cspace`，形成更适合建模与验证的模块边界；
