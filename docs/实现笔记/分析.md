# reL4 capability 子系统现状分析与 `sel4_cspace` 验证边界界定

## 1. 分析目标与范围

主要面向 reL4 中 capability 子系统的当前实现，从现有实现出发，系统梳理 capability 相关逻辑在不同层次中的分布情况，分析各类 capability 操作的调用路径与责任归属，在此基础上界定后续形式化验证的对象边界。

重点分析以下三层中的 capability 相关逻辑：

- `sel4_cspace`
- cnode syscall invocation
- untyped invocation

重点考察以下操作：

 `Copy`
, `Mint`
, `Move`
, `Mutate`
, `Rotate`
, `Delete`
, `Revoke`
, `Retype`

## 2. 代码分布

### 2.1 在 `sel4_cspace` 中

`sel4_cspace` 包含了 capability 核心语义中的主要部分，具体包括：

- capability 派生语义，如 `derive_cap`
- slot 插入、移动、交换等 `cte` 更新操作，如 `cte_insert`、`cte_move`、`cte_swap`
- capability 删除与撤销语义，如 `delete_all`、`revoke`
- capability 派生关系与 `mdb` 维护
- 新对象 capability 的安装，如 `insert_new_cap`
- CSpace 地址解析，如 `resolve_address_bits`

### 2.2 cnode syscall invocation 当前主要负责的内容

`invoke_{cnode, untyped}.rs` 中的各类 capability 操作主要体现为 syscall 层包装逻辑，其职责主要包括：

- 对 syscall 参数进行组织与传递
- 调用前合法性检查
- 对 cap rights、badge/data 等参数进行预处理
- 处理派生失败或非法 cap 等异常情况
- 设置线程状态并组织对 `sel4_cspace` 原语的调用

### 2.3 untyped invocation 当前主要负责的内容

`invoke_untyped.rs` 中与 `Retype` 相关的逻辑主要包括：

- untyped capability 的重置，如 `reset_untyped_cap`
- 目标对象的实际创建，如 `create_object`
- 多个对象的批量创建与安装，如 `create_new_objects`
- 更新 untyped 的 free index 等资源状态
- 在对象创建完成后调用 `insert_new_cap` 将新 capability 安装到目标 slot 中


## 3. 各操作路径分析

### 3.1 调用链路分析表

| 操作 | syscall / invocation 入口 | 中间调用函数 | `sel4_cspace` 原语 | 涉及的数据结构 | 当前主职责层 | 备注 |
|---|---|---|---|---|---|---|
| Copy | `invoke_cnode_copy` | `mask_cap_rights` → `src_slot.derive_cap` | `cte_insert` | `cte_t`、`cap`、`mdb_node`、`seL4_CapRights` | `sel4_cspace` | syscall 层主要做权限裁剪、派生失败检查、错误处理和线程状态设置；真正的 cap 插入与 mdb 维护在 `cte_insert` |
| Mint | `invoke_cnode_mint` | `mask_cap_rights` → `update_data(false, ...)` → `src_slot.derive_cap` | `cte_insert` | `cte_t`、`cap`、`mdb_node`、`seL4_CapRights` | `sel4_cspace` | 比 `Copy` 多了 badge/data 更新；仍由 syscall 层做包装，由 `cte_insert` 完成插入和派生树维护 |
| Move | `invoke_cnode_move` | 读取 `src_slot.capability` 并做空 cap 检查 | `cte_move` | `cte_t`、`cap`、`mdb_node` | `sel4_cspace` | syscall 层只做合法性检查和线程状态切换；移动 slot 与 mdb 重连在 `cte_move` |
| Mutate | `invoke_cnode_mutate` | `update_data(true, ...)` | `cte_move` | `cte_t`、`cap`、`mdb_node` | `sel4_cspace` | 语义上是“更新 cap 数据后移动”；不走 `derive_cap`，直接把新 cap 通过 `cte_move` 放到目标 slot |
| Rotate | `invoke_cnode_rotate` | `update_data(true, ...)` 两次；按 `slot1 == slot3` 分支 | `cte_swap` 或 `cte_move` | `cte_t`、`cap`、`mdb_node` | `sel4_cspace` | syscall 层负责组织三槽位操作；底层状态更新由 `cte_swap` 或两次 `cte_move` 完成 |
| Delete | `invoke_cnode_delete` | `dest_slot.delete_all(true)` | `cte_t::delete_all` | `cte_t`、`cap`、`mdb_node`、`FinaliseCapRet` | `sel4_cspace` | syscall 层几乎只做线程状态设置；删除、finalise、set_empty、派生树摘除都在 `cte.rs` |
| Revoke | `invoke_cnode_revoke` | `dest_slot.revoke()` | `cte_t::revoke` | `cte_t`、`cap`、`mdb_node` | `sel4_cspace` | syscall 层只做包装；`revoke` 在 `cte.rs` 中沿 mdb/派生关系递归删除子能力 |
| Retype | `invoke_untyped_retype` | 可选 `reset_untyped_cap` → `create_new_objects` → `create_object` | `insert_new_cap` | `cte_t`、`cap`、`mdb_node`、`ObjectType`、`PPtr`、untyped cap free index | `untyped + sel4_cspace` | 这是复合操作；对象创建、untyped 空间更新在 untyped invocation，cap 安装与父子关系链接由 `insert_new_cap` 完成 |

### 3.2 调用路径分析结论

从各操作的调用链可以看出，除 `Retype` 之外，`Copy`、`Mint`、`Move`、`Mutate`、`Rotate`、`Delete`、`Revoke` 等操作的核心 capability 状态变化已经集中在 `sel4_cspace` 中完成。`invoke_cnode.rs` 中对应函数主要承担 syscall 层包装职责，而 capability 的插入、移动、删除、撤销以及派生关系维护等核心逻辑已由 `sel4_cspace` 提供实现。

其中，`Copy` 和 `Mint` 的关键在于 capability 派生语义由 `derive_cap` 承担，之后由 `cte_insert` 完成 slot 更新和 `mdb` 维护；`Move`、`Mutate` 和 `Rotate` 的核心在于不同形式的 slot 重排与 capability 更新；`Delete` 和 `Revoke` 的核心则在于 `cte.rs` 中对删除、finalise、zombie 处理与派生关系删除的实现。

`Retype` 则明显不同。它既涉及对象创建和 untyped 资源推进，也涉及新 capability 的安装与父子关系建立，因此是一个横跨 `untyped invocation` 和 `sel4_cspace` 的复合流程。


## 4. 职责分布分析

### 4.1 职责矩阵

| 操作 | 参数检查 | slot 解析 | cap 派生 | slot 更新 | mdb / CDT 维护 | 对象创建 | untyped 更新 | 错误处理 / 调度 |
|---|---|---|---|---|---|---|---|---|
| Copy | `syscall` | `syscall` | `cspace` | `cspace` | `cspace` | - | - | `syscall` |
| Mint | `syscall` | `syscall` | `cspace` | `cspace` | `cspace` | - | - | `syscall` |
| Move | `syscall` | `syscall` | - | `cspace` | `cspace` | - | - | `syscall` |
| Mutate | `syscall` | `syscall` | - | `cspace` | `cspace` | - | - | `syscall` |
| Rotate | `syscall` | `syscall` | - | `cspace` | `cspace` | - | - | `syscall` |
| Delete | `syscall` | `syscall` | - | `cspace` | `cspace` | - | - | `syscall` |
| Revoke | `syscall` | `syscall` | - | `cspace` | `cspace` | - | - | `syscall` |
| Retype | `syscall` | `syscall` | - | `cspace` | `cspace` | `untyped` | `untyped` | `syscall` |

### 4.2 职责矩阵说明

从职责矩阵可以看出，当前 reL4 capability 子系统的职责划分已经具有较强的层次性。

首先，syscall invocation 层主要负责参数检查、slot 解析、错误处理和线程状态调度等外围职责，这些逻辑与系统调用语义密切相关，但不直接构成 capability 核心状态变化本身。

其次，`sel4_cspace` 承担了 capability 核心原语的主要实现职责，包括 capability 派生、slot 更新以及 `mdb` 或派生关系维护等。这说明 `sel4_cspace` 已经具备较明确的 capability 语义承载能力，而不是单纯的底层数据结构操作模块。

最后，`Retype` 中的对象创建和 untyped 状态推进仍然明确属于 untyped 管理层，不宜简单整体归入 `sel4_cspace`。但其中 capability 安装和派生关系维护部分又确实与 cspace 语义直接相关，因此更合适的做法不是把整个 `Retype` 迁入 `sel4_cspace`，而是将其拆分为“对象创建语义”和“capability 安装语义”两个层次来分析。

需要说明的是，`Copy` 和 `Mint` 的 capability 派生由 `derive_cap` 完成，因此其 `cap 派生` 一栏归为 `cspace`；`Mutate` 虽然也会生成新的 capability 数据，但其过程表现为 `update_data + cte_move`，并不经过 `derive_cap`，因此这里不将其单独视为 capability 派生。

## 5. 当前结构特点与验证中的问题

### 5.1 当前结构的特点

与“capability 逻辑高度散乱、核心语义广泛分布在不同模块中”的情形相比，当前 reL4 的 capability 子系统实际上已经具备了较好的模块分层基础。

- capability 核心原语大多已经进入 `sel4_cspace`
- syscall 层主要承担包装和编排职责
- `Retype` 中对象创建与 capability 安装在实现上已有较清晰的分工
- capability 状态变化相关的关键实现可以较自然地收敛到 `sel4_cspace` 这一验证边界

因此，本课题后续工作的重点不应再表述为“从零开始对 capability 子系统重新分层”，而应调整为“在现有较清晰边界上进一步界定验证对象，并围绕 `sel4_cspace` 建立抽象模型与规约”。

### 5.2 验证视角下仍然存在的问题
#### 5.2.1 capability 语义与 syscall 语义仍存在跨层对应关系

虽然 capability 核心状态更新主要位于 `sel4_cspace` 中，但 syscall 层仍负责权限裁剪、参数组织、错误码设置和线程状态切换等逻辑。因而，用户可见操作语义与底层 capability 原语语义之间并是存在“syscall 包装语义到 cspace 原语语义”的映射问题。

这意味着后续验证不能简单只证明底层 `cte` 更新函数的局部正确性，还需要说明 `Copy`、`Mint`、`Move`、`Delete`、`Revoke` 等 syscall 级操作如何由 `sel4_cspace` 原语组合实现。

#### 5.2.2 `Retype` 的验证边界天然跨越两个模块

`Retype` 同时涉及：

- 对象创建
- untyped 资源状态推进
- 新 capability 安装
- 父子派生关系维护

如果不先对 `Retype` 的语义进行拆分，后续抽象建模时就很容易把对象创建语义和 capability 安装语义混杂在一起，导致抽象状态边界不清晰。

#### 5.2.3 当前实现具有较多隐含关系，但缺乏显式抽象模型

`cte_t`、`cap`、`mdb_node` 以及 `delete_all`、`revoke`、`finalise` 等函数共同维护了一组复杂的 capability 状态关系，但这些关系目前主要体现在实现代码本身中，尚未抽象为显式的“状态模型 + 不变式 + 操作规约”。

对于形式化验证而言，仅有代码分层还不够，还需要进一步建立：

- 抽象 slot / cap 模型
- 抽象派生关系模型
- 插入、移动、删除、撤销等操作的抽象语义
- 关键不变式及其保持条件

#### 5.2.4 删除与撤销路径的内部逻辑复杂度较高

`delete_all`、`finalise`、`reduce_zombie`、`set_empty`、`revoke` 等函数构成了 capability 删除与撤销的核心路径，其中又涉及：

- final cap 判断
- zombie capability 处理
- 递归删除
- `mdb` 前后指针重连
- 删除后清理回调

## 后续验证对象划分

### 直接验证对象

后续 Verus 模块级验证可优先围绕以下函数展开：

| 类别 | 函数 / 操作 | 角色 | 是否直接验证 |
|---|---|---|---|
| capability 派生 | `derive_cap` | 生成新 capability 的核心语义 | 是 |
| capability 插入 | `cte_insert` | slot 更新与 `mdb` 维护 | 是 |
| capability 移动 | `cte_move` | slot 更新与关系重连 | 是 |
| capability 交换 | `cte_swap` | 多 slot 重排 | 可选 |
| capability 删除 | `delete_all` | 删除与 finalise 主路径 | 是 |
| capability 撤销 | `revoke` | 删除 descendants | 是 |
| retype 安装 | `insert_new_cap` | 新对象 capability 安装 | 是 |
| 地址解析 | `resolve_address_bits` | CSpace 寻址辅助原语 | 可选 |

### 规约映射对象

以下 syscall 级操作更适合作为上层规约语义，不作为主要逐行验证对象：

| 操作 | 映射关系 |
|---|---|
| `Copy` | `mask_cap_rights + derive_cap + cte_insert` |
| `Mint` | `mask_cap_rights + update_data + derive_cap + cte_insert` |
| `Move` | 空 cap 检查 + `cte_move` |
| `Mutate` | `update_data + cte_move` |
| `Rotate` | 参数组织 + `cte_swap` 或两次 `cte_move` |
| `Delete` | 包装调用 + `delete_all` |
| `Revoke` | 包装调用 + `revoke` |

### `Retype` 的处理原则

`Retype` 应视为复合操作：

- 对象创建与 untyped 状态更新不属于 `sel4_cspace` 核心验证范围
- capability 安装与父子关系维护可纳入 cspace 验证视角
- 因而后续规约设计中，应只抽取 `Retype` 中 cspace 相关语义，而不将整个 `invoke_untyped_retype` 作为单一 capability 原语处理
